## O que é o projeto

SIGES (Siges Serviços Financeiros) é um SaaS de gestão de empréstimos, construído inteiramente por conversa com o Claude Code. Dois papéis: **cliente** (autocadastro, acesso restrito ao próprio contrato) e **gerente/administrador** (acesso total, só é criado internamente por outro gerente — nunca há tela pública de cadastro de gerente). Na UI o papel `gerente` é chamado de "Administrador" — a nomenclatura interna do código (roles, `is_gerente()`, nomes de tabela) continua "gerente" de propósito; não vale a pena renomear (risco alto, zero ganho pro usuário).

Stack: HTML/CSS/JS vanilla sem build/bundler, Supabase (Postgres + Auth + Realtime), Vercel (hospedagem estática + serverless `/api`), GitHub (código-fonte). Detalhes de arquitetura e passo a passo de deploy: [README.md](README.md).

- **Nunca faço `git push` sem confirmação explícita** ("pode enviar", "sim, pode mandar" etc.) — mesmo depois de implementar e testar tudo.
- Quando uma mudança exige rodar uma migration no Supabase, **o usuário roda manualmente** no SQL Editor (não tenho acesso direto ao banco de produção). Só confio que uma função/coluna nova existe em produção depois que ele confirma que rodou a migration.
- Regra especial já usada: se o usuário pedir para eu não interromper o fluxo para pedir aprovações intermediárias, devo implementar tudo primeiro e deixar **uma lista consolidada única** de ações pendentes dele (migrations a rodar, confirmação de push) para o final.
- Sempre testo mudanças de UI no preview antes de reportar como concluído (ver seção de gotchas abaixo sobre cache).
- **Toda vez que um arquivo em `api/*.js` for criado, removido, renomeado, ou tiver a assinatura de `req`/`res` alterada, atualize também o pacote de migração em `netlify/` no mesmo commit** (ver seção "Pacote de migração para Netlify" abaixo) — ele existe pra ficar pronto pra ativar a qualquer momento, e um pacote desatualizado é pior que não ter pacote nenhum (falha silenciosa na hora que mais importa). Mudança de LÓGICA INTERNA de um handler (sem mexer em `req`/`res`/nome do arquivo) não exige tocar em `netlify/` — o wrapper só reexporta o handler original, então já acompanha sozinho.

## Estado das migrations (verifique isto sempre)

`supabase/schema.sql` é a fonte da verdade "fresh install" (sempre mantida em sincronia com a última migration). As migrations em `supabase/migration_00N.sql` são incrementais e devem ser rodadas em ordem crescente, uma vez cada, no SQL Editor do Supabase em produção.

Rode `git status` em `supabase/` para ver quais migrations ainda não foram commitadas (== ainda não confirmadas como rodadas pelo usuário). Migrations não commitadas normalmente significam "ainda não rodei isso em produção" — **nunca assuma que uma coluna/função de uma migration recente já existe em produção sem confirmar com o usuário.**

## Decisões de arquitetura não óbvias (o "porquê")

- **FKs de `notifications_log`, `renewal_cycles`↔`installments`, `payments`, `loan_requests.resulting_contract_id` são todas `ON DELETE SET NULL` ou `CASCADE`** — corrigido depois de um bug recorrente de violação de FK ao excluir contrato/parcela. Havia uma referência circular entre `installments.renewed_into_cycle_id` e `renewal_cycles.origin_installment_id` que tornava impossível qualquer ordem de DELETE funcionar sem esse fix no schema. Se um bug de FK "voltar", o problema quase certamente é uma FK nova que falta esse tratamento — não é reordenar DELETEs em `delete_contract()`.
- **`late_fee_percent`/`late_interest_percent`** (contrato) são aplicados **no momento do recebimento** (quitação OU renovação de parcela/ciclo atrasado), não acumulados automaticamente por um cron. Cálculo: juros de atraso = `saldo_da_parcela × ((1 + late_interest_percent/100)^dias_atraso − 1)` — **juros compostos, taxa DIÁRIA** (decisão explícita do usuário em 2026-07-08, revendo a versão anterior que era simples/mensal — confirmado com exemplo numérico: 1300 a 2%/dia por 8 dias = 1523,16). Multa = `saldo × late_fee_percent/100` (fixa, não escala com dias, sem juros compostos). Ambos os valores aparecem como sugestão editável no modal de recebimento (`gerente-contrato-receber.js`) — o gerente pode ajustar/zerar, tanto na aba "Quitar" quanto na "Renovar". O valor cobrado além do saldo contratual da parcela vira `payments.late_charge_amount`, somado a `interest_component` (assim relatórios/lucro líquido já enxergam automaticamente, sem precisar tocar em cada consumidor). O rótulo do campo é "% ao dia", não "% a.m." — contratos antigos que preencheram esse campo pensando em taxa mensal vão ter o valor reinterpretado como diário (crescimento bem mais agressivo); se isso for um problema, contratos existentes precisam ter `late_interest_percent` ajustado manualmente pelo admin.
- **`refresh_overdue_status()`** (cron diário) recalcula o score de todo cliente com contrato `atrasado`/`perda` a cada execução — sem isso, o score de um cliente inadimplente que não interage mais com o sistema ficaria parado no valor antigo.
- **`payments.net_profit`** é uma coluna gerada (`interest_component - operational_fee_amount`) — é a base de "lucro líquido" em todo o sistema (relatórios, resumo do contrato). Encargo de atraso já está embutido em `interest_component`, então flui automaticamente.
- **Taxa de saída** (`loan_contracts.operational_fee_amount`) é cobrada **uma vez**, na criação do contrato, e soma ao "Total desembolsado" (`total_disbursed_amount`, coluna gerada = `principal_amount + operational_fee_amount`). **Taxa de entrada** (`payments.operational_fee_amount`) é cobrada a cada recebimento. Lucro líquido do contrato = juros recebidos (parcelas + renovações) − taxa de saída (uma vez) − soma das taxas de entrada — já implementado no card "Resumo do contrato" em `gerente-contratos-lista.js` (tela de detalhe).
- **Renovação (`renew_installment`) só é permitida em contratos de parcela única** (`installments_count === 1`) — renovar uma única parcela de um contrato multi-parcela deixaria a relação entre as outras parcelas e o ciclo renovado ambígua.
- **CPF no login**: `doSignIn` detecta se o texto digitado parece CPF (11 dígitos, sem `@`) e resolve para e-mail via RPC `email_for_cpf` antes de chamar `signInWithPassword`.
- **`clients.salary` é `text`, não `numeric`** (mudou na migration_008) — guarda uma das 4 faixas fixas de `INCOME_BRACKETS` (`js/utils.js`), rotulado "Renda Mensal" na UI. Não é usado em nenhum cálculo automático, é só informativo. Clientes cadastrados antes da migration_008 têm o valor numérico antigo salvo como texto (ex: `"3245.00"`) — não bate com nenhuma das 4 opções, então o `<select>` mostra em branco até o admin escolher uma faixa manualmente. Para adicionar/mudar faixas, edite só `INCOME_BRACKETS` em `js/utils.js` — os dois formulários (`login.js` cadastro e `gerente-clientes.js` edição) consomem a mesma constante via `incomeBracketOptionsHtml()`.
- **Inputs `type="number"`/`type="date"`**: `style.css` já remove o spinner nativo e estiliza o ícone do calendário para não destoar do resto do design flat/minimalista — não reintroduza esses controles nativos sem essa camada de CSS.
- **`clients` NÃO tem colunas `cpf`/`phone`** — essas ficam só em `profiles`. Um bug antigo (corrigido na migration_010) tinha `handle_new_user()` tentando inserir `cpf`/`phone` em `clients`, quebrando **todo** cadastro público de cliente com "Database error saving new user" (erro Postgres 42703). Se qualquer função nova mexer em dados de cliente, sempre confira contra o `create table clients (...)` real no `schema.sql` antes de assumir quais colunas existem lá — não pelo nome do campo no formulário.
- **`system_settings` exige `auth.uid() is not null` para SELECT** (corrigido na migration_010) — antes qualquer visitante não-autenticado conseguia ler taxas internas/thresholds direto pela API REST do Supabase (a anon key é pública). Se precisar expor mais uma tabela de configuração, nunca use `using (true)` sem pensar se aquilo devia ser público. (A "Tabela VIP", `loan_rate_reference`, existia com essa mesma proteção mas foi **removida por completo** na migration_013 — era só uma sugestão visual morta, nunca travava nada.)
- **Limite de crédito consome CAPITAL emprestado (`client_outstanding_principal()`), não o saldo devedor total.** `client_outstanding_balance()` (saldo com juros) existe só para exibir dívida ao cliente ("quanto falta pagar") — nunca para calcular/checar limite disponível. Não confundir os dois em código novo: limite de crédito = quanto capital ainda está emprestado; saldo devedor = quanto falta pagar (capital+juros). Bug real já causado por essa confusão: `cliente-dashboard.js` (card "Limite disponível" do Início) usava por engano `client_outstanding_balance`, divergindo do valor correto mostrado em `cliente-solicitar.js` (que sempre usou `client_outstanding_principal`) — corrigido em 2026-07-10. Qualquer tela nova que mostre "limite disponível" deve usar `client_outstanding_principal`.
- **Status "atrasada" de parcela/ciclo tem até 24h de atraso em relação à realidade**, porque só é setado pelo cron diário `refresh_overdue_status()`. Por isso, `gerente-cobrar.js`, `gerente-dashboard.js` e os badges de status (`effectiveInstallmentStatus()` em `js/utils.js`) NUNCA filtram/exibem status de atraso comparando só a coluna `status` — sempre comparam `due_date < hoje` ao vivo. Se adicionar uma tela nova que lista parcelas atrasadas, replique esse padrão (não confie só em `status = 'atrasada'`).
- **`clients` não tem mais `region`** (removida — só existe `client_group`, que agora é uma lista fixa de 5 valores em `CLIENT_GROUPS`, `js/utils.js`). Client group é capturado tanto no cadastro público (`login.js`) quanto na edição pelo admin (`gerente-clientes.js`).
- **Backup/exportação** (`js/screens/backup-export.js`): backup automático roda no máximo 1x por período configurado (`system_settings.backup_frequency`), verificado via `localStorage` (por navegador/dispositivo, não por conta) logo após login do gerente (`maybeRunAutoBackup()` em `onAuthenticated`). Exportação usa SheetJS (XLSX, via CDN) e jsPDF (PDF no estilo do extrato) — nenhuma delas grava nada no servidor, é tudo download direto no navegador.
- **"Planejamento"** (`js/screens/gerente-planejamento.js`, tabela `planning_debts`) é um retrato único ("se eu parasse de emprestar hoje, quanto sobra"), não uma projeção mês a mês — por isso taxa de saída/entrada são aplicadas **uma vez sobre o valor agregado** (Dívida base somada, Faturamento base somado), não parcela a parcela. Dívida base = soma manual de `planning_debts.amount` (dívidas nomeadas, uma data `month` cada, sem limite real de uso — a UI só oferece os próximos 12 meses no seletor). Faturamento base = `system_settings.planning_current_cash` (caixa atual, editado manualmente na própria tela) + soma de **todas** as parcelas/ciclos `pendente`/`atrasada` em aberto no sistema inteiro (sem corte de data — é "tudo que ainda vou receber"). LTV (`system_settings.planning_ltv_percent`) é editado manualmente na mesma tela e desconta do Lucro Bruto pra chegar no Lucro Líquido. Isso é **diferente** do gráfico "Projeção de Lucro (6 meses)" do dashboard (`gerente-dashboard.js`), que É uma projeção mês a mês e aplica a taxa de entrada **por parcela/ciclo individual** (já que ali cada mês soma transações discretas) — não confundir os dois modelos de cálculo ao mexer em qualquer um dos dois.
- **Gráficos de projeção (`areaChartSVG`, `js/charts.js`)**: o rótulo de valor de cada ponto é **sempre desenhado acima do ponto** e em cor fixa (`--ink`, com halo branco atrás via `paint-order:stroke`), nunca na mesma cor da linha/área — o usuário reportou um valor ilegível quando caía em cima de uma linha da mesma cor. Se criar um gráfico de projeção novo, reuse `areaChartSVG` em vez de reinventar rotulagem.
- **Score de crédito (`recalculate_client_score()`)**: fórmula revisada em 2026-07-10/11 (aprovada pelo usuário, com ajustes dele em várias rodadas). **Sistema de "graduação"**: todo cliente novo começa e **permanece com score-base 50** até quitar o primeiro contrato OU pagar a primeira renovação — só nesse marco ("graduado") os BÔNUS de comportamento passam a valer, com piso recentralizado em **70**. **Pós-graduação (2026-07-10, regra mais rígida)**: o bônus (até **+30 pts**) não é mais soma direta de percentuais — é o PRODUTO de `qualidade` (0–1, consistência de pagamento: em dia/adiantado) × `maturidade` (0–1, volume de histórico acumulado — parcelas pagas + contratos quitados + renovações em dia — via `1 - e^(-volume/8)`, retornos decrescentes). Isso torna cada ponto progressivamente mais caro: 1-2 contratos bons chegam a ~80, mas 90-100 exige dezenas de eventos positivos sustentados — impossível de forçar rápido só fazendo contratos em sequência (bug real corrigido: 1 contrato quitado adiantado batia ~100 antes dessa mudança). **Mas as penalidades de risco valem SEMPRE, graduado ou não**: atraso médio histórico (até −20 pts, `v_delay_penalty`), atraso ATUAL — parcela/ciclo vencido e ainda não pago, mesmo padrão "due_date < hoje ao vivo" do resto do sistema (−15 pts fixos, `v_overdue_penalty`), e qualquer contrato em `perda` (−30 pts fixos, `v_perda_penalty`). **Reprovação de solicitação de empréstimo NÃO é critério** (removida por completo — decisão explícita do usuário: "isso jamais deve ser critério de avaliação"). Faixas de tier: Ouro ≥85, Bom 70–84, Atenção 50–69, Alto risco <50 (o limiar 70 é o MESMO usado pra separar as duas listas da tela — ver abaixo). A regra completa NÃO fica mais fixa na tela (poluía o visual) — vive num modal (`openScoreHelpModal()` em `gerente-score.js`), aberto pelo ícone "?" injetado ao lado do título na topbar via `injectScoreHelpButton()` (funciona porque o router seta `topbar-title` ANTES de chamar `render()` — o render pode sobrescrever o innerHTML com o botão, e o próximo `router.handleHashChange()` limpa isso de novo via `textContent =`). Se mudar a fórmula no SQL, atualize esse modal também, senão ficam dessincronizados.
- **Listas "Ranking — melhores scores" / "Atenção — menores scores"** (`gerente-score.js`): são uma **partição estrita por limiar** (`score >= 70` → melhores, `< 70` → piores), nunca mais um "top 10 / bottom 10" por slice — o slice antigo causava um cliente aparecer nas duas listas ao mesmo tempo sempre que havia poucos clientes no total. Se mudar o limiar de graduação (hoje 70), replique a mudança aqui.
- **Dashboard — comparação temporal**: os deltas "▲/▼ X% vs. mês passado" (card de destaque de lucro líquido e "Recebido no mês") comparam o mês corrente **até hoje** com o **mesmo intervalo de dias do mês anterior** (não o mês anterior inteiro) — comparar parcial vs. completo seria injusto (mês em andamento sempre perderia). Lógica em `trendBadgeHtml()` e no cálculo de `prevMonthStart`/`prevMonthEnd` no topo de `renderGerenteDashboard()`.
- **"Top clientes por saldo em aberto"** (dashboard): não usa nenhuma RPC nova — reaproveita as mesmas queries de `installments`/`renewal_cycles` pendentes/atrasados já usadas na projeção de recebimentos, só que com o relacionamento (`loan_contracts!...(client_id, clients!...(profiles!...(full_name)))`) embutido no `select()` pra trazer o nome do cliente junto, e agrupa/soma no JS. Mesmo padrão de embed usado em `gerente-cobrar.js`.
- **`payments.received_at` e `installments.paid_at` são `timestamptz`, mas todas as RPCs de recebimento só gravam uma DATA** (`coalesce(p_received_at, current_date)`, sem hora real) — o Postgres casteia isso pra meia-noite UTC. Formatar com `formatDateTime()` (que usa fuso LOCAL do navegador) desloca pro dia anterior em qualquer fuso negativo (ex: Manaus UTC-4 → 00:00 UTC vira 20:00 do dia anterior). Use sempre `formatDateUTC()` (`js/utils.js`) pra esses dois campos — ela força `timeZone: 'UTC'` no `toLocaleDateString`, recuperando a data exata que foi selecionada. Se adicionar uma tela nova que mostra `received_at`/`paid_at`, replique esse padrão (não use `formatDateTime`/`formatDate` direto neles).
- **Barra de busca perdendo foco a cada tecla**: telas que fazem `root.innerHTML = ...` inteiro dentro do próprio handler de busca (debounced) recriam o `<input>` do zero a cada repaint, matando o foco/cursor. Padrão de correção usado em `gerente-clientes.js` e `gerente-contratos-lista.js`: no início da função de render, capturar `document.activeElement === inputAntigo` e `selectionStart`; depois de montar o novo HTML, se tinha foco, chamar `.focus()` + `.setSelectionRange()` no novo input. `gerente-contrato-novo.js` não sofre disso porque seu `paintClientList()` só substitui a `<div>` da lista, nunca o `<input>` de busca em si — esse é o padrão preferível pra buscas novas (isolar o repaint da lista do repaint do input).
- **Importação em massa de clientes de sistema antigo** (feita uma vez em 2026-07-11, 35 clientes via PDFs do sistema anterior): a técnica foi inserir direto em `auth.users` com senha via `crypt(senha, gen_salt('bf'))` (extensão `pgcrypto`) dentro de um `do $$ ... end $$` por cliente, deixando o trigger `handle_new_user()` já existente criar `profiles`/`clients` a partir do `raw_user_meta_data`. Clientes sem e-mail no sistema antigo receberam um e-mail placeholder (`<cpf-digitos>@sememail.siges.app`) só pra existir uma linha válida em `auth.users` — o login deles continua funcionando normalmente por CPF (via `email_for_cpf`), o placeholder nunca aparece pra eles. Script consumido uma única vez, não faz parte do histórico de migrations numeradas (é dado, não schema) — se precisar importar mais gente no futuro, reusar esse mesmo padrão.
- **Texto de cobrança via WhatsApp** (`buildWhatsappUrl()` em `gerente-cobrar.js`) inclui uma estimativa de juros de atraso + multa (mesma fórmula composta diária do modal de recebimento) quando a parcela/ciclo já está vencido — é só uma SUGESTÃO no texto, o valor final cobrado continua sendo o que o gerente ajustar na hora de receber de verdade.
- **PDF de "Exportar dados"** (`runExportPDF()` em `backup-export.js`) é organizado por CLIENTE (nome → contratos → parcelas em aberto de cada contrato), não mais em 4 tabelas soltas sem relação entre si. A seção "Pagamentos" no fim continua como lista flat separada (histórico, não "em aberto") — se pedirem pra também agrupar pagamentos por cliente, é uma extensão do mesmo padrão (`contractsByClient`/`openInstallmentsByContract` já prontos pra reusar a lógica).
- **Menu mobile "Mais"** (`#mobile-more-menu` + `#mobile-more-backdrop` em `index.html`, lógica em `main.js`): a tabbar mobile só cabe ~5 itens legíveis, mas o gerente tem 10 rotas no menu. `MOBILE_TAB_ROUTES.gerente` define só as 4 mais usadas como aba direta (Dashboard/Cobrar/Contratos/Relatórios); o resto vira uma folha (`toggleMobileMoreMenu()`) que abre por cima da tabbar com grid 3 colunas. Fecha automaticamente ao navegar (via `closeMobileMoreMenu()`, já chamado em todo `router.handleHashChange()`) ou ao clicar no backdrop. Cliente só tem 5 rotas então cabe inteiro sem "Mais".
- **Hash da URL preso entre sessões**: `doSignIn`/`doSignUp`/`handleSignOut` agora fazem `location.hash = ''` antes de autenticar/deslogar — sem isso, se o usuário saía estando em `#/cliente/emprestimos`, o hash ficava na URL e o próximo login caía direto naquela tela em vez do Dashboard (`onAuthenticated()` só força Dashboard quando o hash já está vazio). Qualquer novo fluxo de auth (ex: um terceiro provedor OAuth) precisa do mesmo cuidado.
- **Transferência de score do sistema antigo** (feita uma vez, 2026-07-11): é só um valor inicial "cosmético" — a regra de graduação do Siges recalcula o score de qualquer cliente no primeiro evento real dele (pagamento, quitação, ou um clique em "Recalcular todos"), e como os clientes importados começam sem nenhum histórico de pagamento NO SIGES, esse recálculo os leva de volta a 50 (não graduados). Não é uma transferência permanente de histórico de crédito, só evita começar todo mundo igual visualmente no dia da importação.
- **Senha do cliente (e agora do gerente) NÃO é visível pro admin, nem pode ser** — Supabase Auth guarda senha como hash bcrypt irreversível, não existe "ver a senha atual/trocada pelo usuário". Em vez disso, o admin **redefine** a senha em `gerente-clientes.js`/`gerente-gerentes.js` (seção "Redefinir senha" dentro do modal de editar, só no modo edição) → `POST /api/reset-client-password` (mesmo endpoint serve os dois casos, apesar do nome — service_role, `PUT /auth/v1/admin/users/{id}` com `{password}}`) → nova senha fica visível só no campo que o próprio admin acabou de digitar (repassa manualmente). `/api/reset-client-password.js` e `/api/update-user-email.js` (2026-07-11) agora checam o papel do ALVO via `getTargetProfile()` (`api/_lib/supabaseAdmin.js`): se o alvo é `role='gerente'`, exige `caller.is_primary_admin` — senão um gerente secundário conseguiria resetar a senha/e-mail de outro administrador (inclusive o primário) e sequestrar a conta. Clientes continuam abertos pra qualquer gerente ativo mexer.
- **Trilha de auditoria** (`audit_log` + RPC `log_audit_event(p_action, p_description, p_metadata)`, tela `gerente-auditoria.js`, entre Planejamento e Configurações no menu): callable por `anon` E `authenticated` de propósito — precisa registrar `login_falho`/`login_sucesso` mesmo SEM sessão totalmente estabelecida (usa `auth.uid()`, que é `null` no caso de falha; `actor_name`/`actor_role` viram "Anônimo"/`null`). RLS só permite SELECT pra `is_gerente()`; não existe policy de INSERT — a função é `security definer` e roda como dono da tabela, bypassa RLS (mesmo padrão de toda função `security definer` do schema). Helper JS `logAudit(action, description, metadata)` em `state.js`, chamado após sucesso em: criar/editar/excluir cliente, aprovar/rejeitar cadastro, redefinir senha, criar/editar/excluir contrato, editar parcela, receber pagamento, renovar, criar/editar gerente, e login com/sem sucesso. **Login bem-sucedido** é capturado no listener `supa.auth.onAuthStateChange` de `auth.js`, filtrando especificamente `event === 'SIGNED_IN'` (dispara só em login de verdade — senha ou Google) e nunca em `'INITIAL_SESSION'` (restauração de sessão ao recarregar a página) — não confundir os dois eventos se mexer nesse listener. ## Cuidado com `create or replace function` e parâmetros

**`CREATE OR REPLACE FUNCTION` só substitui a função existente se o nome E os TIPOS dos parâmetros forem idênticos aos já cadastrados.** Se você mudar o TIPO de um parâmetro existente (ex: `p_salary numeric` → `p_salary text`) ou mudar a ARIDADE (adicionar/remover parâmetro), o Postgres cria uma **segunda função sobrecarregada (overload)** em vez de substituir — as duas ficam coexistindo no banco.

Isso já causou um bug real (migration_008 mudou `update_client_profile`'s `p_salary` de `numeric` para `text`, sem trocar nome/quantidade de parâmetros — resultado: duas versões idênticas em nome e nomes-de-parâmetro, e o PostgREST não conseguia mais decidir qual usar, dando "Could not choose the best candidate function" em todo salvamento de cliente pelo admin). Corrigido na migration_010 com `drop function if exists <nome>(<tipos antigos exatos>);` antes do `create or replace`.

**Regra prática:** toda vez que uma migration mudar o TIPO de um parâmetro existente ou adicionar/remover parâmetro de uma função já em produção, inclua um `drop function if exists nome(tipos-antigos-exatos-na-ordem);` ANTES do `create or replace function`, pegando os tipos antigos exatos do `git show <commit-anterior>:supabase/schema.sql`.

## Gotchas de teste/verificação

- **Service Worker cacheia agressivamente.** Se algo que você editou parece "não ter efeito" no preview, ou o browser mostra uma versão antiga do JS mesmo após reload, rode isto no console/`preview_eval` antes de qualquer outra investigação:
  ```js
  (async () => {
    const regs = await navigator.serviceWorker.getRegistrations();
    for (const r of regs) await r.unregister();
    const keys = await caches.keys();
    for (const k of keys) await caches.delete(k);
    location.reload();
  })();
  ```
  Isso já causou pelo menos um "bug" reportado pelo usuário que na verdade era cache — sempre descarte essa hipótese antes de mexer em código que já parece correto na leitura.
- **Sempre suba `CACHE_NAME` em `sw.js`** ao final de uma rodada com mudanças relevantes de JS/CSS (está em `v15` agora) — isso força o Service Worker do usuário em produção a invalidar o cache antigo no próximo load.
- Para testar telas de gerente no preview sem depender de dados reais do Supabase (RLS bloqueia sessão fake), mocke `supa.from` com um proxy "chainable" que resolve para dados fake, teste, e restaure `supa.from` original antes de terminar. Às vezes só desregistrar o Service Worker + limpar caches não basta pro `preview_eval` pegar o JS novo (o servidor estático do preview também cacheia o `<script src>` de cada arquivo individualmente, mesmo depois de um hard navigate com cache-bust na URL da página) — se uma variável/função continuar "antiga" depois do procedimento padrão, faça `fetch(caminho, {cache:'no-store'}).then(r=>r.text())` e `(0, eval)(texto)` pra aplicar o conteúdo atual na sessão do browser. **Atualização (2026-07-11): às vezes até `{cache:'no-store'}` sozinho continua servindo conteúdo velho** — só resolve de fato acrescentando um cache-buster na URL do fetch (`fetch(caminho + '?v=' + Date.now(), {cache:'no-store'})`). Se mesmo assim uma função global (ex: `router.handleHashChange`) continuar com o código antigo (teste rápido: `algumaFuncao.toString().includes('trechoNovo')`), o hard-navigate comum não bastou — chame `navigate` com `{force:true}` pra forçar um reload de verdade da página inteira.
- **`(0, eval)` não funciona em arquivos com `const`/`let` no escopo top-level que já rodaram uma vez na página** (ex: `js/main.js`, que declara `const NAV_ITEMS`/`const MOBILE_TAB_ROUTES`; `js/router.js`, que declara `const ROUTES`/`const router`) — reavaliar o texto do arquivo gera `SyntaxError: Identifier ... has already been declared`, que aborta o `eval` inteiro em silêncio (a Promise só rejeita, nada aparece no retorno se você não tratar o erro) e as variáveis global continuam com o valor antigo. Pra esses arquivos específicos, rode o texto dentro de `new Function(texto + '; suaFuncaoDeRender();')()` — isso executa num escopo de função isolado (sem colidir com os `const` já existentes no global) e ainda deixa você chamar a função de render logo em seguida pra inspecionar o DOM resultante. **Isso NÃO serve pra "hot-patchar" um singleton como `router`** — o `router`/`ROUTES` recriados dentro do `new Function` isolado ficam desconectados do `router` global que o resto do app já referencia (closures antigas de `hashchange` continuam presas na versão velha); pra esses dois arquivos específicos, é mais simples forçar reload completo (`navigate` com `force:true`) do que tentar hot-patch. Arquivos de tela normais (`js/screens/*.js`) só usam `function` no top-level, que PODE ser redeclarada sem erro — `(0, eval)` funciona neles direto.

## Auditoria de pagamento parcial (2026-07-12, `migration_022`)

Rodada de testes dirigidos a cenários de recebimento parcial de parcela — 8 cenários rastreados manualmente (linha a linha do SQL), 2 bugs reais encontrados e corrigidos nesta migration, 2 achados baixos também corrigidos:

- **`receive_cycle_payment()` (quitar um ciclo de renovação) não validava o valor recebido contra o devido** — diferente de `receive_payment()` (parcela normal), que já bloqueia valores fora do permitido tanto no JS quanto no SQL. Um valor MENOR que o `full_debt_amount` do ciclo era aceito do mesmo jeito, o ciclo (e o contrato, se fosse a última pendência) era marcado como quitado, e os componentes gravados em `payments` (`principal_component`/`interest_component`) eram o valor CHEIO esperado, não o que realmente veio em `p_amount_received` — ou seja, o sistema registrava lucro que não entrou de verdade. **Ciclo de renovação não tem colunas de controle de pagamento parcial** (diferente de `installments`, que tem `principal_paid_partial`/`interest_paid_partial`) — a correção escolhida (das duas possíveis) foi exigir que o valor recebido seja igual ao valor cheio devido (`full_debt_amount + encargo de atraso`, tolerância de 1 centavo), rejeitando qualquer coisa fora disso com `INVALID_AMOUNT`. Reforçado também no JS (`gerente-contrato-receber.js`) com uma mensagem amigável explicando que ciclos não suportam parcial e sugerindo usar "Renovar" se o cliente só puder pagar uma parte.
- **`renew_installment()`, ao renovar direto de uma PARCELA, ignorava pagamento parcial já recebido** — lia `principal_share`/`interest_share` brutos (os valores originais do contrato) em vez de descontar `principal_paid_partial`/`interest_paid_partial` já pagos. Se uma parcela recebia pagamento parcial e depois era renovada (em vez de continuar sendo quitada), o valor já pago "sumia": o novo ciclo nascia cobrando a dívida CHEIA original, inflando o saldo devedor do cliente exatamente pelo valor que ele já tinha pago. Corrigido subtraindo `principal_paid_partial`/`interest_paid_partial` no `select` que lê a parcela (mesmo ajuste que `receive_payment` já fazia corretamente), com `greatest(0, ...)` como defesa extra contra qualquer estado inconsistente anterior. O branch que renova a partir de um CICLO já existente (não de uma parcela) não tinha esse problema — `renewal_cycles` não tem coluna de pagamento parcial, então `full_debt_amount` já representa o saldo remanescente real.
- **`update_installment_schedule()` (editar/reagendar parcela) não validava contra o que já foi pago** — deixava reduzir `principal_share`/`interest_share` pra um valor abaixo de `principal_paid_partial`/`interest_paid_partial` já registrados, o que geraria saldo restante negativo (`amount_due - paid_partial < 0`) e um "Pago parcial: resta R$-X" confuso nas telas do cliente. Agora rejeita com `AMOUNT_BELOW_ALREADY_PAID` (mapeado pra mensagem amigável no modal), e o próprio modal mostra um aviso em vermelho com o valor já pago quando a parcela editada já tem pagamento parcial.
- **Confirmado correto e não alterado**: alocação juros-primeiro-depois-capital em pagamento parcial de parcela normal; acúmulo correto de múltiplos pagamentos parciais na mesma parcela; `client_outstanding_balance()`/`client_outstanding_principal()` já descontavam pagamento parcial de parcela normal; bloqueio de valor inválido (zero/negativo/acima do permitido) em `receive_payment`; proteção contra corrida (`for update` + recheck de status) em todas as funções de recebimento; parcela parcialmente paga nunca conta como "paga" no score (comportamento intencional).

## Bug real corrigido: exibição de ciclos renovados nas telas do cliente (2026-07-12, `migration_023`)

Reportado pelo usuário com print real: contrato de parcela única renovado 3x mostrava as linhas intermediárias ("Renovação 1"/"Renovação 2") com um traço no lugar do badge de status, e o valor de TODAS as linhas de renovação mostrava a dívida cheia (ex: R$650) em vez do que foi de fato pago naquela renovação (só juros, ex: R$150).

- **Causa raiz**: `cliente-emprestimos.js`/`cliente-indicacoes.js` tinham dois problemas no `cyc.map(...)` que renderiza a tabela de parcelas/ciclos: (1) o mapa de labels de status (`{ pendente: 'Pendente', paga: 'Paga', atrasada: 'Atrasada' }`) não incluía `renovada` — resultado `undefined`, badge vazio (o traço visto no print); (2) a coluna "Valor" sempre mostrava `r.full_debt_amount` (a dívida CHEIA que rola pro próximo ciclo), nunca `r.interest_only_amount` (o que foi realmente pago NESSA renovação — só juros). **A tela de admin (`gerente-contratos-lista.js`, card "Ciclos de renovação") nunca teve esse bug** — já mostra as duas colunas separadas ("Juros pago" = `interest_only_amount`, "Dívida renovada" = `full_debt_amount`) e já tinha `renovada` no mapa de status; o bug era exclusivo das 2 telas do cliente.
- **Correção**: `renovada` adicionado ao mapa de status nas duas telas. Valor da linha agora é condicional: `status === 'renovada'` → mostra `interest_only_amount` (o juros pago nessa renovação específica); qualquer outro status (`pendente`/`atrasada`/`paga`) → mostra `full_debt_amount` como antes (pra um ciclo ainda aberto, isso é corretamente "quanto falta pagar"; pra um ciclo `paga`, é corretamente "quanto foi pago na quitação final", já que `receive_cycle_payment` só aceita o valor cheio — ver achado anterior).
- **Melhoria de nomenclatura pedida junto**: a última linha de um contrato renovado (status `paga`, quitação final) mostrava "Renovação N" como as outras — agora mostra **"Quitação"**, já que semanticamente não é mais uma renovação, é o encerramento da dívida.

## Análise de continuidade: formato de backup e portabilidade de hospedagem (2026-07-13)

Pedido do usuário: analisar (sem implementar ainda) os dois riscos mais sérios já levantados numa auditoria anterior — perda de dados e disponibilidade da Vercel. Entregue como artifact único (favicon 🧭, "análise inicial") cobrindo os dois temas. Nenhuma mudança de código nesta rodada — só análise + recomendação, seguindo a convenção do projeto de propor antes de implementar decisões arquiteturais.

- **Backup (.json)**: confirmado por leitura de `collectAllSystemData()` (`js/screens/backup-export.js`) que o backup diário cobre 6 tabelas (clientes/contratos/parcelas/ciclos/pagamentos/solicitações) mas **não** inclui `system_settings`, `planning_debts`, `audit_log`, `notifications_log`, `push_subscriptions` nem perfis de gerente. **Não existe nenhuma função de importação/restauração em lugar nenhum do código** — o app só exporta. Resposta direta à pergunta do usuário ("consigo restaurar com o .json?"): hoje, não, sem escrever um script novo na hora. Isso não é um problema do formato JSON em si (ele preserva os dados perfeitamente) — é a ausência de um caminho de restauração pronto. Senha de login de cliente nunca está em nenhum backup, em nenhum formato (limitação do Supabase Auth, hash irreversível) — restauração real sempre vai exigir resetar senha de cada cliente depois, formato-independente. **Recomendação dada (não implementada)**: manter o `.json` como está E adicionar um segundo arquivo `.sql` (dump com `INSERT`s em ordem de FK) ao mesmo backup automático diário — porque cola direto no SQL Editor do Supabase, a mesma ferramenta que o usuário já usa toda semana pras migrations, sem precisar de script de importação customizado no momento do aperto. Sugerido também guardar o arquivo num .zip protegido por senha (tem CPF/telefone/renda em texto puro). Se aprovado numa rodada futura, é um `runBackupSQL()` novo ao lado de `runBackupJSON()` em `backup-export.js`, chamado junto em `maybeRunAutoBackup()`.
- **Portabilidade de hospedagem**: auditoria de todos os `api/*.js` + `vercel.json` confirmou que o lock-in real na Vercel é **baixo** — zero imports `@vercel/*`, zero uso de Vercel KV/Blob/Edge Config, segredos são `process.env` comum, front-end sem build/bundler (já é o mais portátil possível). O único arquivo genuinamente específico da Vercel é `vercel.json` (config do cron, 3 linhas) — a lógica de negócio dos 7 endpoints não muda numa migração, só a "casca" que recebe a requisição. **Lovable explicado e descartado como opção**: é um gerador de app via IA com hospedagem própria (Lovable Cloud) — usá-lo significaria reconstruir o SIGES do zero dentro da ferramenta deles, incompatível com a exigência explícita do usuário de que só a plataforma de hospedagem mude, mantendo função/interface/identidade visual intactas. **Plataformas recomendadas, em ordem**: Netlify (Plano B — grátis permite uso comercial, funções Node quase idênticas, cron nativo, troca de menor atrito), Render (Plano C — Node.js completo, os 7 arquivos praticamente inalterados, só embrulhados num servidor Express pequeno), Cloudflare Pages+Workers (Plano D — melhor custo/performance de longo prazo, mas motor diferente da Vercel, exigiria reescrever de fato a assinatura das 7 funções — não é resposta de emergência). Nenhum arquivo de configuração de outra plataforma foi criado nesta rodada (evitar manutenção de config pra plataforma não usada) — oferecido como próximo passo opcional se o usuário quiser o "pacote de migração" pronto pro Netlify especificamente.

Depois da análise de continuidade (seção acima), o usuário aprovou as duas propostas e pediu implementação direta, com o pacote de migração mantido sempre atualizado.

### Backup automático agora baixa dois arquivos

`js/screens/backup-export.js`: `runBackupSQL(silent, preloadedData)` gera um dump com `insert into ... values (...) on conflict (<PK>) do nothing;` para as mesmas 6 tabelas que já saíam no `.json` (`clients`, `loan_contracts`, `installments`, `renewal_cycles`, `payments`, `loan_requests`), na mesma ordem FK-safe que `collectAllSystemData()` já usava. `buildSqlDump(data)` monta o texto; `sqlLiteral(v)` faz a serialização por tipo (string escapada com `''`, número como está, boolean `true`/`false`, `null`→`NULL`, objeto/array→JSON stringificado). A coluna `profiles` embutida (join só pra exibição) é excluída das colunas de `clients` — não é uma coluna real da tabela.

- **`maybeRunAutoBackup()`** agora busca os dados uma vez (`collectAllSystemData()`) e reaproveita pros dois formatos (`runBackupJSON(true, data)` + `runBackupSQL(true, data)`), com um `setTimeout` de 600ms entre os dois downloads — disparar dois arquivos no mesmíssimo instante às vezes faz o navegador tratar o segundo como pop-up e bloquear silenciosamente.
- **Configurações → Backup e exportação** ganhou um segundo botão manual, "Fazer backup agora (.sql)", ao lado do já existente "(.json)".
- **O `.sql` gerado sempre abre com um comentário de aviso** explicando o pré-requisito: as contas de autenticação (`auth.users`/`profiles`) de cada cliente precisam já existir no banco de destino com os mesmos IDs antes de rodar os `INSERT`s — o Supabase Auth guarda senha como hash irreversível, então nenhum backup feito pelo navegador (sem `service_role`) consegue recriar contas sozinho. Isso não é um bug do dump, é a mesma limitação estrutural já documentada em outros lugares deste arquivo — só deixei explícito no próprio arquivo baixado, pra quem for usar num momento de aperto não ter falsa expectativa.
- **Não implementado de propósito nesta rodada**: um restaurador automático de contas (`auth.users`) a partir do backup. Recriar contas em massa é possível (mesma técnica já usada na importação de 2026-07-11, documentada acima), mas é uma ferramenta *separada*, que precisaria de `service_role` (não pode rodar no navegador) — ficou fora do escopo do que foi pedido ("o segundo download em .sql"). Se quiser esse restaurador completo no futuro, é a extensão natural: um script/endpoint novo que lê o `.sql`/`.json`, recria cada `auth.users` com senha temporária, e deixa o `handle_new_user()` cuidar do resto.

### Pacote de migração para Netlify (`netlify.toml` + `netlify/`)

Dormente enquanto o site roda na Vercel — não afeta o deploy atual em nada (a Vercel ignora `netlify.toml`, e o `vercel.json` continua sendo a configuração ativa). Existe pra reduzir o tempo de uma migração de emergência a "ativar o site no Netlify", não a "reescrever o backend".

- **`netlify/functions/_adapter.mjs`**: `adaptVercelHandler(vercelHandler)` — traduz a assinatura clássica da Vercel (`(req,res)` com `res.status(x).json(y)`) pro formato que o Netlify Functions espera (`(event) => ({statusCode,headers,body})`). Não reimplementa nenhuma regra de negócio — só a "casca".
- **`netlify/functions/<nome>.mjs`** (7 arquivos, um por endpoint: `create-user`, `delete-client`, `reset-client-password`, `update-user-email`, `wipe-all-data`, `notify-event`, `cron-daily-check`) — cada um tem 3 linhas: importa o handler ORIGINAL de `api/<nome>.js` (sem copiar/colar lógica) e reexporta via `adaptVercelHandler`. `cron-daily-check.mjs` também exporta `config = { schedule: '0 10 * * *' }` (convenção atual do Netlify Scheduled Functions, mesmo cron do `vercel.json`).
- **`netlify.toml`**: `publish = "."` (mesma raiz estática de sempre, sem build), `functions = "netlify/functions"`, e um redirect `/api/* → /.netlify/functions/:splat`. Esse redirect é o que permite que **nenhuma linha do frontend precise mudar** — confirmado por grep que todo `fetch(...)` do app usa caminho relativo (`fetch('/api/create-user', ...)`, nunca uma URL absoluta da Vercel), então o redirect resolve tudo por baixo dos panos.
- **`netlify/README.md`**: só um ponteiro pra esta seção do CLAUDE.md (evita duas fontes de verdade documentando a mesma coisa).
- **Não verificado contra uma conta Netlify real** (sem acesso nesta sessão pra testar deploy de verdade) — a lógica do adaptador foi revisada manualmente linha a linha contra os 7 handlers reais (confirmado por grep que nenhum usa `res.send`/`res.end`/`res.setHeader`, só `res.status().json()`, que é 100% do que o adaptador cobre) e testada com dados sintéticos no preview, mas o formato exato do `config.schedule` e o shape de `event.headers`/`event.body` devem ser confirmados na documentação do Netlify **no momento real da migração**, já que plataformas mudam essas convenções com o tempo.
- **Regra permanente**: ver bullet novo em "Como o usuário trabalha comigo" — qualquer `api/*.js` novo/removido/renomeado, ou com assinatura de `req`/`res` alterada, precisa do wrapper correspondente atualizado em `netlify/functions/` no mesmo commit.

## Limitações conhecidas (v1, ver README para detalhes)

- **E-mail para clientes está desativado de fato (decisão consciente, 2026-07-07).** O usuário não tem domínio próprio registrado (só tentou cadastrar `siges.com.br` no Resend sem possuir o domínio de verdade — verificação trava em "Not Started" porque não há onde adicionar os registros DNS). Decisão: não registrar domínio por enquanto; os canais reais de notificação do cliente são o **sino in-app** (Supabase Realtime) e o **Web Push** (ambos gratuitos, já funcionando). `RESEND_FROM_EMAIL` continua sem valor em produção, então todo envio cai no remetente sandbox `onboarding@resend.dev`, que só entrega para o e-mail da própria conta Resend — **isso é esperado, não é bug**. Email e push são canais independentes em `dispatchToRecipient` (`api/notify-event.js`), então a falha de e-mail não afeta a entrega do push. Se o usuário decidir registrar um domínio no futuro, o caminho é: Resend → Domains → verificar DNS → configurar `RESEND_FROM_EMAIL` no Vercel — nenhuma mudança de código é necessária.
- WhatsApp é só via link `wa.me` (zero custo, sem API paga) — não há envio automático de WhatsApp pelo servidor.
- Sem paginação em nenhuma listagem (`gerente-clientes.js`, `gerente-contratos-lista.js` etc.) — inofensivo no volume atual, mas é o primeiro ponto a atacar se o negócio crescer muito.
- Sem testes automatizados — valide sempre manualmente no preview, incluindo viewport mobile.

## Onde procurar o quê

- `supabase/schema.sql` — schema completo "fresh install", toda regra de negócio sensível vive aqui como função SQL (`security definer`), nunca só em JS.
- `js/screens/*.js` — uma tela por arquivo, registradas via `registerRoute(...)` no fim de cada arquivo. `js/main.js` é sempre o último `<script>` carregado.
- `js/charts.js` — gráficos SVG inline reutilizáveis (`barChartSVG`, `lineChartSVG`, `donutChartSVG` + `donutLegendHtml`).
- `api/*.js` — únicas rotas que usam a `service_role` key ou outras chaves secretas (Resend, VAPID, CRON_SECRET).
